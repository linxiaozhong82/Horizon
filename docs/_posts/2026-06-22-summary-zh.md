## [宁可重复，不要错误的抽象（2016）](https://sandimetz.com/blog/2016/1/20/the-wrong-abstraction) ⭐️ 8.0/10

Sandi Metz 在 2016 年的博文中主张，过早或不正确的抽象比代码重复危害更大，敦促开发者容忍重复，直到出现清晰、可复用的模式。 这一原则已成为现代软件工程的基石，影响了重构实践，并对 DRY（不要重复自己）原则的教条式应用提出了挑战。 文章强调错误的抽象会造成远程耦合和维护噩梦，而重复是可逆的，以后更容易重构。

hackernews · rafaepta · 6月21日 16:08 · [社区讨论](https://news.ycombinator.com/item?id=48620090)

**背景**: 在软件开发中，DRY 原则鼓励通过将共享代码提取到抽象中来避免代码重复。然而，如果抽象设计不当或过早应用，可能导致代码僵化且难以修改。Sandi Metz 的建议是允许重复，直到某个模式充分重复，从而证明一个干净的抽象是合理的。

**社区讨论**: 评论者大多赞同该文章，分享了过度工程化代码库的个人经验。一些人讨论了“单一真相来源”原则，而另一些人指出函数式编程减少了重复问题。普遍情绪是，工程不足的代码比过度工程的代码更容易修复。

**标签**: `#software engineering`, `#abstraction`, `#code duplication`, `#refactoring`, `#programming practices`

## [开发者误解 CORS，文章指出问题](https://fosterelli.co/developers-dont-understand-cors) ⭐️ 8.0/10

一篇 2019 年的博客文章指出，许多开发者误解了 CORS（跨源资源共享），将其与安全边界混淆，而没有认识到它是一种请求伪造预防机制。该文章因高互动量和富有洞见的社区更正而重新引起关注。 CORS 是一种关键的 Web 安全机制，但普遍的误解会导致配置错误和安全漏洞。这一讨论凸显了开发者需要加强对 Web 安全基础知识的理解。 文章特别指出，设置 Access-Control-Allow-Origin 并不能阻止其他源发送请求，它只控制浏览器是否允许 JavaScript 读取响应。社区评论进一步澄清，CORS 并非安全边界，而是同源策略的放松机制。

hackernews · toilet · 6月21日 01:35 · [社区讨论](https://news.ycombinator.com/item?id=48614844)

**背景**: CORS 是一种浏览器机制，允许网页向提供页面的源之外的其它源请求资源。它是为了放松同源策略而发展出来的，同源策略默认会阻止跨源请求以保障安全。预检请求（使用 OPTIONS 方法）会在非简单请求时发送，用于检查服务器权限。混淆通常源于 CORS 被错误地视为服务器端访问控制，而不是浏览器强制执行的协调协议。

## [请求反对而非许可：沟通策略革新](https://www.mooreds.com/wordpress/archives/3518) ⭐️ 7.0/10

文章建议，专业人士不应请求许可，而应告知经理或团队自己的计划，并设定截止日期邀请反对意见。这从寻求批准转变为事先告知，除非被告知不行，否则默认承担责任。 这种方法能通过减少反复沟通来提升效率，并展现主动性。它在尊重组织层级的同时，赋予员工在其职责范围内行动的权力。 该策略最适合在个人职责范围内的行动；对于超出范围的任务，仍需请求许可。文章强调设定明确的反对截止日期，以避免无限期拖延。

hackernews · skogstokig · 6月21日 20:17 · [社区讨论](https://news.ycombinator.com/item?id=48622234)

**背景**: 在许多工作场所，员工习惯在行动前请求许可，这会拖慢进度并造成不必要的审批循环。“请求反对”技巧将沟通重构为带有退出条款的主动通知，类似于“默认执行”方法。这一概念与敏捷和创业文化中的“行动偏向”理念相关。

**社区讨论**: 社区成员反应不一：有人认为这赋予员工权力，将责任转移到行动者身上；另有人警告，若在没有适当权限时使用，可能被视为越界，且具体措辞很重要——原型与生产部署截然不同。总体而言，该建议被认为有用，但需视情境而定。
